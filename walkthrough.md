## 📚 2026-07-03 零系统开销的按需动态“图片自愈”与后台异步解析系统 (On-Demand Dynamic Image Healing & Background Resolution)

为了彻底修复“可能有其他人仔零件或整体实物无法显示”的现象，我们设计并上线了一套**按需触发（Lazy / On-demand）的全站图片自愈修补网**。它能够在**不产生任何服务器启动遍历开销与被反爬封禁风险**的前提下，随着用户的每一次浏览自动修补全库所有缺失实物图的人仔：

### 1. 💡 为什么不采用“全库启动遍历扫描”？
* 我们的 SQLite 数据库中包含 **17,000+** 个人仔。如果让服务器在后台遍历检索这 1.7 万个图像链接，不仅会消耗极其庞大的带宽和数小时的漫长时间，更是会因为向 Brickset/BrickLink 集中发起过万次并发请求而瞬间被对方反爬系统永久封禁 IP。

### 2. 📂 后端按需图像解析接口 (`/api/resolve-image`)
* 我们新建了专门用于修复图像的 API 接口 `/api/resolve-image`：
  1. 它只接收前端在“大图加载失败”时发来的修复请求；
  2. 后端接收后，从本地提取出该人仔的英文原名（如 `"Toad - Scared"`），并瞬时到 Brickset 人仔目录中检索，获取官方的 BrickLink ID（如 `mar0096`）；

### 1. 🤖 巨型人仔 (Big Fig) 拼搭逻辑与爆炸图对齐
* **缺陷原因**：
  * Big Fig（如 Killow [fig-002816](file:///Users/xiaozhi/Desktop/find/app.js#L709)）没有独立的 legs（腿部，通常分类是61），其身体与腿部是一体化巨型躯干（`Body Giant`，分类为13）。
  * 原先的 Pass 2 模糊匹配中，手臂/牛角/触手等附属零件因名称带有 `Body`（如 `Animal Body Part`）被错误识别为 Torso，导致真正的巨型身躯被抢占；而在 Pass 3 兜底中，巨型身躯又因为含有 `Pin`（在 `3 Pin Holes` 中）被 `isSpuriousPart` 误杀，最终导致双臂或运动头盔被错配至 Torso / Legs 槽位，显示出拼搭错位。
* **修复与优化**：
  * **增设巨型一体化身躯前置匹配 (Step 0)**：在零件分类第一步之前，直接检查是否存在分类 ID 为 `13`（Body Giant）的零件，一旦存在，直接锁定为 `torsoPart`，且强制设置 `legsPart` 为 `null`（代表一体化巨型下肢无需拼装）。
  * **增强 Spurious 排除规则**：
    * 改进 `isSpuriousPart` 对 `pin` 的判定规则为 `(name.includes('pin') && !name.includes('hole'))`，避免含销孔的大型主体身躯被误判排除。
    * 将 `animal body part`（如牛角、尾巴）显式加入 spurious 过滤列表，防止其抢占躯干词条。
  * **爆炸图层自适应中心悬空对齐**：
    * 当加载 Big Fig 时，给组装舞台 `.lego-figure-canvas` 动态挂载 `.is-bigfig` 类，通过 CSS `display: none !important` 隐藏 Legs 零件图层（防止空缺时显示问号占位符）。
    * 在 CSS 中，为 Big Fig 剩下的 3 个图层（Hat, Head, Torso）量身定制了纵向 translate 的坐标偏移（Hat at -100px, Head at -20px, Torso at 60px），保持完美的 80px 等距纵向爆炸分布，视觉上极为对称。
